# ML Experiment Reviewer Environment

## Introduction

_Everyone says AI is going to replace software engineers—but can it actually debug the invisible logic of a broken ML pipeline?_

The ML Experiment Reviewer is an RL environment designed to evaluate how well AI agents can review, debug, and optimize Machine Learning experiments. Finding silent bugs in ML pipelines is a challenging engineering task. Instead of basic code generation, this environment procedurally generates flawed ML experiments and requires the agent to diagnose the root causes. By testing agents across varied architectures, data splits, and hyperparameter configurations, we evaluate their debugging capabilities under constraints.

## The Three Difficulty Tiers

The environment progresses from basic pattern matching to numerical deduction:

- **Easy (The Basics):** The agent receives fundamental metrics including Training Accuracy, Validation Accuracy, and the overall loss progression across epochs. By carefully analyzing how these learning curves diverge, the agent must identify whether the model is collapsing into Overfitting, stalling due to Underfitting, or generalizing effectively as a Good Fit.
- **Medium (The Architect Test):** The agent receives a pipeline configuration with healthy hyperparameters. The core task focuses on finding logic errors, such as using the wrong model architecture for the data type, spotting class imbalances, or detecting biased train/test splits.
- **Hard (The Math Test):** The pipeline architecture is flawless, and the agent only receives raw numbers: hardware constraints (e.g., GPU Memory) alongside a full list of low-level hyperparameters (Learning Rate, Batch Size, Epochs, Dropout, Weight Decay, Optimizer, and LR Scheduler). The agent must mathematically calculate if the numbers safely interact with one another and the hardware, such as realizing a massive batch size of 512 will instantly crash a low-memory 8GB GPU.

## Grading System

Agents are graded on precision to avoid rewarding keyword stuffing.

- **Strict Output Arrays:** The agent must return a specific array of issues found rather than unstructured text.
- **Proportional Rewards:** Points are awarded based on identifying the true, hidden bugs generated by the environment.
- **False Alarm Penalty:** For every bug the AI claims exists that does not actually exist in the experiment state, it loses 10% of its base score. This structurally penalizes random guessing.

## Experimental Observations

While benchmarking this environment using Qwen-72B, we made the following observations regarding LLM behavior:

- **Verbosity in structured tasks:** The model frequently generated extensive explanations rather than concise bug lists, attempting to cover all potential ML issues despite strict schema requirements.
- **Hallucinations under constraint:** In the Hard Task, the JSON payload only provides hyperparameters with zero dataset information. Despite this, the LLM repeatedly hallucinated categorical errors, frequently citing "severe class imbalance" as an issue.
- **Scoring impact:** Because the model hallucinated categorical errors not present in the environment logic, the false alarm penalty mathematically reduced its final score to near 0%. This highlights a specific struggle with pure numeric deduction when narrative contextual clues are removed.

## Baseline Benchmark Scores

We evaluated Qwen-72B across 30 games per tier to establish an initial performance baseline:

- **Easy Score:** ~84% (Strong at recognizing standard Overfitting/Underfitting metrics)
- **Medium Score:** ~27% (Struggled to map architectures correctly without introducing false positives)
- **Hard Score:** ~0% (Failed at numeric deduction; heavily penalized for hallucination)

## Quick Start: Example Interaction

Here is an example of what the Agent sees and how it must respond during a Medium tier task.

**Agent Input (Observation Payload):**

```json
{
  "task_difficulty": "medium",
  "experiment_data": {
    "data_type": "csv",
    "model": "Vision Transformer (ViT)",
    "class_distribution": { "majority_class": 0.45 },
    "applied_transforms": ["Tokenization"],
    "train_val_split": 0.8
  }
}
```

**Expected Agent Output (Action Schema):**

```json
{
  "diagnosis": "The architecture and preprocessing do not match the data type.",
  "issues_found": ["wrong_model_architecture", "incorrect_preprocessing"],
  "suggestions": [
    "Switch to XGBoost or a Random Forest for CSV data",
    "Remove text Tokenization"
  ],
  "reason": "Vision Transformers are for image data, not tabular CSV data. Tokenization is meant for text.",
  "data_quality": "Balanced distribution",
  "preprocessing": "Applied text transforms to tabular data",
  "model_check": "Wrong model for tabular data",
  "overall_score": 0.2
}
```

_(Note: The `overall_score` is a field where the agent provides its own subjective assessment of the experiment's health between 0.0 and 1.0. This is independent of the environment's actual backend grading mechanism)._
